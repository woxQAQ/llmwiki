# 工程技术：在智能体优先的世界中利用 Codex

## 摘要

- 这篇 OpenAI 工程博客总结了一个极端实验: 一个内部产品从空仓库起步, 在约五个月内扩展到约一百万行代码, 团队坚持不手写任何产品代码, 主要由 Codex 智能体完成实现、测试、文档、CI、工具和修复。
- 作者将工程工作的重心定义为环境设计、意图表达、约束编码和反馈回路建设, 而不是直接写代码。
- 文中核心主张是, 在智能体优先的软件工程中, 代码仓库本身应成为可版本化、可索引、可机械验证的记录系统, 让智能体能直接从仓库中读取架构、规范、计划和产品知识。
- 文章也强调, 高吞吐量智能体系统需要新的治理方式, 包括短生命周期 PR、自定义 lint 和结构测试、后台文档维护与垃圾回收式重构。

## 关键信息

- 团队以 `Codex CLI + GPT-5` 为起点, 让智能体从模板生成初始仓库结构、CI、格式化规则和首版 `AGENTS.md`。
- 人类通过提示驱动任务, 让 Codex 打开 PR、调用审查智能体、响应反馈并迭代到可合并状态。
- 为了让智能体可以直接执行 QA, 团队让应用、日志、指标和浏览器状态都对 Codex 可读, 包括 git worktree 启动、Chrome DevTools 协议、DOM 快照、截图、LogQL 和 PromQL。
- 团队放弃超长 `AGENTS.md`, 转向短地图加结构化 `docs/` 知识库, 并用 lint/CI 验证文档是否更新、交叉链接和与代码一致。
- 他们优先强制执行架构边界和品味不变式, 例如层级依赖方向、边界解析、结构化日志、命名规范和文件大小限制, 而不是微观规定每个实现细节。
- 作者明确提醒, 这些能力高度依赖专门投入的仓库结构与工具, 不应简单假设可以零成本泛化。

## 值得记录的证据

- 文中声称团队以约人工写代码十分之一的时间, 构建并交付一个已有内部日活和外部 alpha 用户的软件产品。
- 文章给出的仓库规模约为一百万行代码, 五个月内约 1,500 个 PR 由三人小团队推动, 后续扩到七人时吞吐量继续上升。
- 作者将失败经验归因于情境管理问题, 明确反对把大量说明塞进单一 `AGENTS.md`, 并提出 "给 Codex 一张地图, 而不是一本 1000 页说明书"。
- 文中描述 Codex 已能在该仓库中端到端执行复现 bug、录制演示、实施修复、验证、开 PR、处理反馈、修复构建故障并合并变更。
- 文章同时指出完全自主会复制既有坏模式, 因此需要把黄金原则编码到仓库里, 并持续运行后台任务清理偏差和技术债。

## 开放问题

- 这些结论主要来自 OpenAI 内部单一产品团队的经验总结, 缺少可复现实验和外部案例对照。
- 文中没有系统量化成本、失败率、返工率和长期维护成本, 因此很难判断这种模式在其他团队中的总拥有成本。
- 作者承认尚不清楚完全由智能体生成的系统在更长时间尺度上的架构连贯性会如何演化。
